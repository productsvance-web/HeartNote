**Acceptance criteria** (full template per `.claude/rules/acceptance-criteria.md`):

*Engineering*:
- Plan stated and approved before any code (this doc).
- No new tables. No new severity values. No new symptom enum values.
- Diff scoped to manual-entry surface; no drive-by refactors elsewhere.

*Functional happy path*:
- When the caregiver opens `/log/manual` on a fresh today, all 5 cards render with no controls pre-selected; Save button disabled until at least one control is touched.
- Tapping `Mild` on Swelling → Save: a new `daily_log_symptom_events` row exists with `symptom='swelling', present=true, severity=1` and `source_log_id` = today's daily_log id (created if not present).
- Tapping the Weight stepper to `182.4` → Save: a new `daily_log_readings` row exists with `field='weight_lb', value=182.4`.
- Tapping `2` on Pillows → Save: today's `daily_logs.pillow_count = 2`.
- Tapping `At rest / can't finish sentences` on Breathing → Save: `daily_log_symptom_events` row with `severity=4`, AND the engine fires `tier_1_911` on next dashboard render (verify via `daily_assessments.tier`).

*Edge cases*:
- Caregiver opens form, makes no changes, taps Save → no rows written, no error, returns to dashboard.
- Caregiver fills only Pillows → only Pillows writes (per Conflict 4).
- Caregiver dictated this morning, opens manual entry tonight, fills Swelling=Mod → only the new symptom_event row appended; this morning's voice-extracted rows untouched.
- Today's `daily_log` doesn't exist yet → form save creates one with `processing_status='complete'` and uses its id as `source_log_id`.
- Today's `daily_log` exists in `processing_status='pending'` (a voice log mid-processing) → manual save fails closed with "Voice log still processing — try again in a moment" rather than racing.

*Error states*:
- Network failure on Save → form preserves the user's selections, shows "Couldn't save — try again".
- Out-of-range value (weight 800) → Zod rejects at server-action boundary, returns field-level error.
- Auth expired mid-session → server action redirects to `/login` (existing pattern).
- RLS rejects the insert (somehow another patient's id passed) → 500 with structured log; user sees generic error.

*Performance*:
- Page renders in ≤2s on 4G (server fetches snapshot+baseline; both already cached patterns).
- Save action returns in ≤1s for the typical 1–5 row write.

*Persistence*:
- All writes go through Supabase server client per CLAUDE.md rule #2.
- RLS policies on `daily_log_readings`, `daily_log_symptom_events`, `daily_logs` are already in place from earlier migrations — verify they cover the manual-entry insert path before wiring.

*Permissions/RLS*:
- The RLS policies on the three target tables already allow `caregiver_id = auth.uid()` writes via patient ownership. Verify by reading the migration file before code.

*Side effects*:
- Engine re-evaluation runs (same trigger as voice-log save) — `daily_assessments` row created/updated for today.
- Dashboard auto-refreshes via Next.js's revalidatePath('/dashboard').

*Manual verification*:
